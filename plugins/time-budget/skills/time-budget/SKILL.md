# time-budget — 능동 시간 측정

## 전제 (설계상 사실)
Claude에겐 경과 시간을 감지하는 센서가 없다. "시간을 느껴라"는 못 지킨다.
대신 **능동 조회**와 **측정 가능한 프록시 최소화**로 바꾼다.

## 되는 메커니즘 (검증됨)
1. **`date` 능동 프로브** — Bash로 벽시계를 조회한다. 작업 시작에 stamp, 결정 지점마다 elapsed 확인.
   - `clock.sh start <label>` / `clock.sh elapsed <label>` (이 스킬 폴더의 헬퍼; 출력은 초)
   - 헬퍼 없이도 `date +%s` 두 번 빼면 됨.
2. **백그라운드 트립와이어 = 알람** — `sleep N; echo "TRIPWIRE <label> ${N}s"` 를 Bash `run_in_background:true`로 띄운다.
   백그라운드 완료 시 하니스가 Claude를 `<task-notification>`으로 **재호출** → "N초 경과" 알람이 된다.
   본 작업이 먼저 끝나면 무시(TaskStop 또는 그냥 방치). "외부 작업(빌드/배포/서버)이 N초 넘으면 알려줘"에 정확히 맞음.
   (근거: 백그라운드 명령 완료가 실제로 재호출을 일으킴 — 세션 내 관찰됨.)
3. **장기 작업은 `run_in_background`로** — 블로킹 대신 백그라운드 → 완료 시 종료코드와 함께 통지. 기다리는 동안 다른 일 가능 + 사후에 실제 소요를 앎.

## 안 되는 것 (한계 — 정직)
- **턴 내부의 자기 생성은 어떤 타이머로도 중간에 못 끊는다.** 13KB base64 벽·긴 회고를 생성하기 시작하면 멈출 수 없다.
- 알람/트립와이어는 **턴 사이 / 도구 완료 시에만** 울린다. 즉 타이머는 ①행동 전 예측 ②외부 장기작업 감시 ③턴 간 경과 체크에만 유효.
- 폭주 생성은 타이머가 아니라 **프록시 규율**로만 막는다(아래).
- haiku 워치독 서브에이전트는 **비추천**: 서브에이전트도 시계가 없고 Claude 내부 상태를 못 본다. 결국 `sleep N`과 동급이라 백그라운드 sleep이 더 싸다.

## 시간 프록시 최소화 (타이머가 못 잡는 부분)
사용자 체감시간과 직결되는, 행동 전에 셀 수 있는 지표들. 이걸 줄이는 게 실질 규율:
- **왕복 횟수** — 결과까지 도구 호출 몇 번? 합쳐서 줄여라(독립 호출은 한 메시지에 병렬).
- **빈 턴 = 0** — "지금 ~합니다"로 턴을 끝내지 마라. 선언과 tool call을 **같은 메시지**에. 빈 턴은 사용자 시간 1회 순손실.
- **출력량** — base64 벽·장문 회고 생성 금지. 생성 자체가 기다리는 사람의 실제 시간.
- **우회 단계** — 없는 문제를 위해 파이프라인 발명 금지. 사용자가 더 짧은 길을 주면(예: "파일로 떨궈", "plain string") 즉시 그 길로, 내 절차로 회귀 금지.

## 절차 (긴 작업 시작 시)
1. `clock.sh start <task>` — T0 stamp.
2. 예산 잡기: 결과까지 왕복 N회 예상. 외부 장기작업이 있으면 트립와이어 `sleep <budget>` 백그라운드로.
3. 결정 지점마다 `clock.sh elapsed <task>` 로 경과 확인. 예산 초과면 멈추고 접근을 바꾼다(같은 길로 덧칠 금지).
4. 끝나면 트립와이어 정리(TaskStop) — 안 그러면 나중에 깜짝 알람.
