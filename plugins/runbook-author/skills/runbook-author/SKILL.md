# runbook-author — 환경 결합 런북 스킬 저작법

한 세션에서 지난한 시행착오 끝에 성공한 절차(UI QA·E2E·인프라 조작 등)를, **다음 세션이 재발견 없이
그대로 재사용**할 수 있는 런북 스킬로 박제하는 방법론. 관찰된 문제에서 출발한다: CDP/UI 자동화 같은
작업은 한번 성공한 세션은 계속 성공하지만, 새 세션은 같은 시행착오를 처음부터 반복한다 —
세션 간 상속되는 건 기억이 아니라 기록뿐이므로, 기록의 형태가 재사용성을 결정한다.

## 핵심 원리 — 무엇이 희소 자산인가

일반론("브라우저를 열고 버튼을 클릭한다")은 모델이 이미 안다. 스킬로 남길 가치가 있는 것은
**이 환경에서 실제로 통과한 구체**뿐이다:

1. **성공 코드/명령 원문** — 요약·의사코드 금지, 통과한 스니펫 그대로. 가능하면 **실패한 변형과의
   차이를 주석으로**(예: "plain object headers 병합은 실패 — 반드시 Headers 객체"). 실패 변형 병기가
   다음 세션의 최단 디버깅 경로다.
2. **실측 절차** — 낡으면 깨지는 값(포트·PID·활성 프로필·현역 워크트리)은 값을 박제하지 말고
   **그 값을 알아내는 명령**을 박제한다. 환경이 바뀌어도 절차는 드리프트를 감지해준다.
3. **함정 목록 (사고 출처 명시)** — 실제 사고에서 나온 금지 수칙만. "그룹 토글을 스크립트로 클릭하다
   삭제 사고 → evaluate 클릭은 '적용'/'저장' 정확 텍스트만" 처럼 **사건 → 수칙** 형태로. 출처 없는
   일반 주의사항은 노이즈다.
4. **판정 기준선 (날짜 명시)** — "무엇이 정상인가"의 해석 규칙. 예: "vision abort WARN 은 무해",
   "LLM 플랫 산출 ~40%는 모델 변동성이지 파이프라인 실패가 아님". 셀렉터보다 오래 살아남고,
   이게 없으면 다음 세션이 정상 신호를 버그로 오진하며 시간을 태운다. 기준선엔 측정 날짜를 박는다.
5. **예산** — 대기 상한(예: 생성 정착 240s), 재시도 횟수(1회) 같은 수치. 근거는 인프라 한계가 아니라
   제품 요구사항으로. 상한 도달 = 판정이지 연장 신호가 아니다.

공식: **낡으면 깨지는 값은 실측 절차로, 낡지 않는 것(코드 원문·함정·판정 규칙)은 그대로 박제.**

## 작성 시점 — 성공 직후, 그 세션 안에서

런북은 반드시 **성공한 바로 그 세션이** 쓴다. 그 세션만이 무엇이 실패했고 무엇이 통과했는지의
전체 diff 를 갖고 있다. 나중에 회고로 쓰면 실패 변형·사고 맥락이 소실되고 일반론으로 퇴화한다.
캠페인(반복 작업)이라면 첫 1~2회 성공 직후 초안을 쓰고, 캠페인이 끝나며 함정·기준선을 증보한다.

## 구조 템플릿

```markdown
# <이름> — <한 줄: 무엇을 어디서 자동화하는가>
<전제 요약 + "실측 검증된 레시피, YYYY-MM-DD 캠페인에서 전 구간 성공" 같은 검증 이력 1줄>

## 0. 전제 스택 점검 (순서대로, 실측으로만)
   — 라우팅/프로세스/세션/자격증명 각각: 무엇을 어떤 명령으로 실측하는지. 위임할 에이전트가 있으면 명시.
## 1. 레시피 (검증된 순서)
   — 단계별로 성공 코드 원문. 실패 변형은 ⚠ 주석으로 병기.
## 2. 함정 (전부 실사고에서 나옴)
   — 사건 → 수칙. 출처 없는 항목은 넣지 않는다.
## 3. 판정 기준선 (YYYY-MM-DD 기준)
   — 정상/비정상 해석 규칙, 알려진 변동성, 오진 방지 규칙.
```

## 2층 구조 — 발견 경로까지 설계해야 완성

스킬 파일만 만들면 다음 세션이 그 존재를 모른다. 반드시 두 층을 함께 둔다:

- **스킬(전문)**: 절차·코드·함정 전부. 필요할 때만 로드되므로 길어도 된다.
- **메모리 포인터(인덱스)**: 매 세션 자동 로드되는 메모리 인덱스에 한 줄 —
  "이 작업의 SSOT 는 스킬 X 다, 재발견 금지". 절차 전문을 메모리에 넣지 않는다(용량·중복).

## 위임 규격

대량 반복(N회 캠페인)은 서브에이전트에 위임하게 되는데, 이때 스킬의 §레시피·§함정을
**프롬프트에 원문 첨부**할 수 있는 형태로 써 둔다 — "스킬을 참고해라"가 아니라 복붙 가능한 절차로.
안전 수칙(파괴적 조작 금지 목록)은 위임 프롬프트의 첫 섹션에 반드시 포함시킨다.

## 무엇을 스킬로 만들지 판단

- ✅ 이 환경 고유 + 재발견 비용 큼 + 반복 가능성 있음 (UI E2E, 배포 검증, 복잡한 로컬 스택 기동)
- ❌ 모델이 이미 아는 일반론, 일회성 작업, 레포 안에 이미 문서화된 것(중복 금지 — 포인터만)

## 유지 — 검증하며 상속

런북도 정전(canon)이다: 다음 세션은 **경배가 아니라 검증**으로 상속한다. §0 실측이 어긋나면
맹신하지 말고 스킬을 현재에 맞게 갱신하고, 낡은 함정·기준선은 날짜와 함께 은퇴시킨다.
갱신도 성공 직후 그 세션이 한다.
